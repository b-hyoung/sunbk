접근 방식 — 실제 사진 통합 전략

정리된 요구사항 기반으로 3가지 접근 방식을 비교합니다

확정된 방향: 가상 선박 → 실제 선박 교체 / 파일명 리네임 후 public/images/vessels/ 배치 / 선박 상세 갤러리 확장 + 독립 작업현장 페이지 (/testN/work)
A

선박 중심 구조 (추천)

사진을 선박별로 그룹화하여 데이터 구조에 녹임

vessels.json 개편
실제 선박 8척(수연1/3/5/6/9호, 신성호, 영진호, 진양2호) 데이터로 교체.
vessel_images 배열에 부위별 사진(외관, 조타실, 기관실, 선실, 주기관 등) 다중 등록
vessels/[slug] 상세 페이지
대표 이미지 1장 → 부위별 사진 갤러리(그리드 + 라이트박스)로 확장
/testN/work 작업현장 페이지 (신규)
모든 선박의 작업/정비/항해 사진을 모아 타임라인 또는 카테고리별 갤러리
about 페이지
기존 작업사진 갤러리는 /work 페이지로 이관. about은 회사소개에 집중

장점

  • 선박 데이터와 사진이 한 곳에서 관리됨
  • 선박 상세 페이지가 실질적인 정보 제공
  • Supabase 전환 시 구조 그대로 유지 가능

단점

  • vessels.json 전면 재작성 필요
  • 파일명 → 선박/부위 매핑 작업 필요
B

사진 중심 구조

사진을 별도 JSON(work-photos.json)으로 관리하고 선박 데이터와 느슨하게 연결

vessels.json 개편
실제 선박으로 교체하되 대표 이미지 1장만 등록
work-photos.json 확장
모든 사진을 여기서 관리. 선박ID/카테고리(외관/정비/항해)/날짜 메타데이터 포함
vessels/[slug] 상세 페이지
work-photos.json에서 해당 선박 사진을 필터링하여 갤러리 표시
/testN/work 작업현장 페이지
work-photos.json 전체를 카테고리/선박별 필터로 표시

장점

  • 사진 추가/삭제가 vessels.json과 독립적
  • work 페이지와 상세 페이지가 같은 소스 공유

단점

  • 두 개의 JSON을 동기화해야 함
  • Supabase 전환 시 별도 테이블 필요
C

하이브리드 (A + B 혼합)

선박 대표/부위 사진은 vessel_images에, 작업현장 사진은 work-photos.json에 분리

vessels.json
실제 선박 + vessel_images에 선박 자체 사진(외관, 조타실, 선실 등)
work-photos.json
작업/정비/항해 등 액션 사진만 별도 관리 (상가 작업, 환경정화, 안전교육 등)
페이지 구성
상세 페이지 → vessel_images / work 페이지 → work-photos.json

장점

  • 데이터 성격에 맞는 분리
  • 각 페이지가 명확한 소스를 가짐

단점

  • 사진 분류 기준이 모호할 수 있음 (기관실 사진 = 선박 사진? 작업 사진?)
  • 구현 복잡도가 가장 높음
추천: A (선박 중심 구조) — 사진 대부분이 특정 선박에 귀속되는 구조이고, vessel_images 배열이 이미 존재하므로 자연스러운 확장. work 페이지에서는 모든 선박의 vessel_images를 합쳐서 보여주면 별도 JSON 없이도 가능.
